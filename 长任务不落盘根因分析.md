# 长任务模型不落盘写入根因分析

## 问题描述

执行长任务（如创建 PRD）时，模型反复在"思考→规划→再思考"中循环，始终不会真正调用文件写入工具落盘。

---

## 执行链路概览

```
用户请求
  → agentRouting.mjs: 路由判定 (default/model-planning/state-graph/planner)
    → agent.mjs: runDefaultAgent()
      → runModelPlanningGate(): 先调一次模型做计划
        → modelPlanningRuntime.mjs: 返回 JSON plan (steps[])
          → runSingleStep(): 逐步执行
            → 每个 step 又是一次完整 LLM 调用
              → 模型返回 content (文本) + tool_calls
                → executeToolCalls(): 执行工具
                  → enforceEvidencePolicy(): 证据门控
                    → 判定 completionState
                      → 确定是否继续下一步
```

---

## 根因分析

### 根因 1: Planning Gate 让模型产出"计划"而非"行动"

**位置**: `agent.mjs` `runModelPlanningGate()` (~line 3126) + `modelPlanningRuntime.mjs`

Planning Gate 在执行任何工具之前，**先调一次模型做计划**。系统提示明确要求：

> "First decide whether the latest request is a new task or a continuation... Output a JSON plan with steps..."

这意味着对于"创建 PRD"这种任务，模型的第一反应是**规划步骤**（理解需求→调研→撰写→验证），而不是直接动手写文件。

**关键问题**: 计划中的每个 step 都会被 `normalizeDefaultStepPlan()` 转化为独立的 LLM 调用。每个 step 的模型调用又会经历同样的 thinking → planning → 再思考 循环。

**相关代码**:
- `agent.mjs:3126` - `runModelPlanningGate()` 入口
- `agent.mjs:3154` - `normalizeDefaultStepPlan(planningGate)` 将计划规范化
- `modelPlanningRuntime.mjs:1-120` - Planning 系统提示

### 根因 2: 默认路由把复杂任务降级为 `default` 模式

**位置**: `agentRouting.mjs`

路由逻辑中，`default` 模式是最通用但也最"保守"的模式。它不会强制模型使用特定工具，而是让模型自主决定。对于 PRD 这类任务：

- 不会路由到 `model-planning`（需要显式 planning 触发条件）
- 不会路由到 `planner`（需要 planner 专用流程）
- 落入 `default` → `runDefaultAgent()` → 模型自主决策 → 倾向于"先想想再做"

### 根因 3: 步骤级 LLM 调用缺乏"必须落盘"的强制约束

**位置**: `agent.mjs` `runSingleStep()` 系列函数

每个 step 的执行是一次独立的 LLM 调用。模型在每一步都可以：
1. 返回纯文本 content（不调工具）→ 等于"思考输出"
2. 返回 tool_calls 但调的是 `search_code`/`read_file`（只读操作）→ 等于"还在调研"
3. 真正调用 `write_file`/`apply_patch` → 才是落盘

没有任何机制**强制**模型在某个步骤必须产出写入操作。模型完全可以每一步都返回文本或只读工具调用。

### 根因 4: 证据门控（Evidence Policy）对"有产出"的定义过于宽松

**位置**: `agentEvidence.mjs` `enforceEvidencePolicy()` (line 642) + `collectEvidenceFromToolEvents()` + `deriveCompletionState()`

证据门控检查 `toolEvents` 来判断任务是否完成。但它只关心：
- `hasToolEvents` - 是否有工具调用事件
- `hasApprovalBlock` - 是否有审批阻塞
- `hasCapabilityBlock` - 是否有能力阻塞

**不检查**：
- 工具调用类型（只读 vs 写入）
- 是否有持久化产出（文件创建/修改）
- 产出质量是否匹配任务目标

这意味着模型调了 `read_file` 3 次，证据门控就认为"有执行证据"，可以标记步骤完成。然后进入下一步，下一步又可以只读。

### 根因 5: 步骤完成判断与"继续下一步"的循环逻辑

**位置**: `agent.mjs` 主循环 + `completionGate.mjs`

主循环的继续条件是：
1. `completionState` 不是 `completed` / `blocked`
2. 还有剩余步骤
3. 没有超过最大步骤数

`completionGate.mjs` 的 `deriveCompletionState()` 判断逻辑：
- 如果 `hasToolEvents === true` 且没有 blocker → 步骤标记为"完成"
- 然后继续下一步

这形成了一个**安全但低效的循环**：
```
Step 1: 模型思考 + read_file × 2 → evidence 满足 → 标记完成 → 继续
Step 2: 模型思考 + search_code × 1 → evidence 满足 → 标记完成 → 继续
Step 3: 模型思考 + read_file × 1 → evidence 满足 → 标记完成 → 继续
...
直到 maxSteps 耗尽或模型碰巧决定写文件
```

### 根因 6: Planning Prompt 中 `context` 类步骤的陷阱

**位置**: `modelPlanningRuntime.mjs` 的计划分类

Planning 系统提示允许模型输出 `context` 类型的步骤（纯信息收集/思考），这类步骤**不需要工具调用**。模型在不确定如何执行时，会倾向于把任务拆成多个 `context` 步骤，每个步骤只是"思考和整理"。

对于 PRD 这种任务，模型可能产出：
```json
{
  "steps": [
    { "type": "context", "description": "理解产品需求和目标用户" },
    { "type": "context", "description": "调研竞品和市场定位" },
    { "type": "context", "description": "整理功能需求列表" },
    { "type": "execute", "description": "撰写 PRD 文档" }
  ]
}
```

前 3 步都是纯思考，只有最后一步才可能触发写入。但如果前面步骤消耗了太多 token/轮次，最后一步可能永远不会被执行到。

## 核心结论

**模型不落盘的根本原因是：整个执行框架的"成功标准"是"有工具调用事件"，而不是"有持久化产出"。** 这让模型可以在每一步都安全地只做只读操作（思考、搜索、读取），而框架认为这是"正常执行"并继续推进。

具体表现为三层叠加：

1. **Planning 层**: 让模型先规划，规划本身就是"思考"
2. **执行层**: 每步独立 LLM 调用，没有强制产出约束
3. **验证层**: 证据门控不区分只读和写入，只要"有动作"就算完成

---

## 修复方向

### 方向 A: 证据门控增加"写入要求"（最小改动）

在 `enforceEvidencePolicy()` 中，对 `answerMode === 'execute'` 的任务，检查 `toolEvents` 中是否包含写入类工具调用（`write_file`、`apply_patch`、`edit_file`、`create_artifact`）。如果没有写入证据，步骤不算完成。

### 方向 B: Planning Prompt 强制"执行优先"（中等改动）

修改 `modelPlanningRuntime.mjs` 的系统提示：
- 禁止纯 `context` 类步骤，或限制 `context` 步骤不超过 1 个
- 要求计划中至少有 1 个 `execute` 步骤涉及文件写入
- 在步骤描述中明确"产出物必须落盘"

### 方向 C: 步骤级强制产出检查（较大改动）

在 `runSingleStep()` 的返回逻辑中，增加检查：
- 如果 step 类型是 `execute`，但本轮 LLM 调用没有任何写入类工具调用 → 标记为未完成，要求重试
- 设置最大重试次数（如 2 次），超过后强制模型进入写入模式

### 方向 D: 路由层识别"产出型任务"（较大改动）

在 `agentRouting.mjs` 中，识别"需要产出文件"的任务（PRD、报告、文档等），路由到专门的执行模式，该模式：
- 跳过 Planning Gate 或简化为 1 步
- 第一步就要求模型调用 `write_file` 创建文件骨架
- 后续步骤基于已有文件迭代

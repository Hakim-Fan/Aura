# Agent 商业化进阶架构方案设计报告

## 1. 概述
针对当前 Agent 存在的“工具过载”和“决策逻辑僵化”问题，本报告提供两套成熟的商业级解决方案。这些方案旨在从底层逻辑上解决 Agent 对工具的盲目依赖，提升响应速度与用户体验。

---

## 2. 方案 A：智能意图路由与动态工具注入 (Smart Routing & Dynamic Injection)
**核心思想：** 在模型感知到工具之前，通过确定性的逻辑或轻量级预判，物理上限制工具的暴露范围。

### 2.1 技术实现组件
1.  **意图网关 (Intent Gateway)**:
    - 在 `runAgent` 入口处，调用极速小模型（如 Gemini Flash）对用户意图进行 3-5 类分类。
    - 类目示例：`GENERAL_KNOWLEDGE` (通用百科), `WORKSPACE_ACTION` (文件操作), `WEB_RESEARCH` (网页搜索)。
2.  **物理过滤器 (Tool Filter)**:
    - 若判定为 `GENERAL_KNOWLEDGE`，在向 LLM 传递 `tools` 数组时直接置空或仅保留文本相关插件。
    - 消除模型“为了用工具而用工具”的心理暗示。
3.  **软性证据政策 (Soft Evidence Policy)**:
    - 废除“无工具调用即失败”的截断逻辑。
    - 引入 **内容质量评估器**：检查回答中是否包含对应的解决方案代码、步骤描述。如果质量达标，即便未执行工具，也视为任务完成。

### 2.2 商业优势
- **极速响应**：避免了重型工具（如浏览器）不必要的启动时间。
- **成本优化**：减少了无效的工具调用产生的 Token 和计算开销。
- **稳定性**：通过确定性的路由减少了模型的随机性。

---

## 3. 方案 B：规划-执行-分析 编排架构 (Plan-Execute-Review Orchestrator)
**核心思想：** 将 Agent 从单一的“对话者”重构为“任务调度系统”，分阶段处理复杂需求。

### 3.1 技术实现流程
1.  **规划层 (Planner)**:
    - 模型第一轮交互仅允许输出 `Plan`（JSON 格式）。
    - Plan 必须包含步骤描述和“是否需要外部工具”的布尔值。
2.  **控制层 (Controller)**:
    - 由系统代码解析 Plan。
    - 如果所有步骤的“外部工具”均为 `false`，则立即通过 LLM 生成直接回答，流程终止。
3.  **审查层 (Reviewer)**:
    - 在工具执行完后，由另一组 Prompt 或模型检查：工具输出是否解决了用户问题。
    - 如果已解决，则不再进行后续的搜索，防止“翻来覆去搜个没完”。

### 3.2 商业优势
- **极高的确定性**：适用于严谨的工程、自动化测试场景。
- **任务可见性**：用户可以清晰看到“思考中 -> 规划中 -> 执行中”的每一步进度。
- **长链条任务处理能力**：能够处理需要多点协作的复杂任务。

---

## 4. 方案对比与落地建议

| 维度 | 方案 A (智能路由) | 方案 B (全任务编排) |
| :--- | :--- | :--- |
| **首字延迟 (TTFT)** | **最低 (优秀)** | 较高 (需先生成计划) |
| **首选场景** | 桌面端助手、编程辅助、日常问答 | 自动化运维、复杂代码大规模重构 |
| **可维护性** | 高 (逻辑清晰) | 中 (状态机复杂度高) |
| **商业案例** | **GitHub Copilot, Cursor** | **GPT-Engineer, Devin** |

### 5. 针对本项目 (Desk-Agent) 的落地优先级策略
鉴于当前项目是桌面端侧工具，**强烈建议优先实施方案 A**：

1.  **Step 1:** 重构 `capabilitySelector.mjs`。引入基于语义的“轻量意图分类”，对百科类问题直接屏蔽浏览器工具。
2.  **Step 2:** 优化 `agent.mjs` 中的 `enforceEvidencePolicy`。增加对回复内容的启发式评估（Heuristics），不要一刀切地截断无工具调用。
3.  **Step 3:** 调整 System Prompt。将“强制性指令”改为“条件触发指令”。

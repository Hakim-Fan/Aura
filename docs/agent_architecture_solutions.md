# Agent 商业化进阶架构方案设计报告

## 1. 概述
针对当前 Agent 存在的“工具过载”和“决策逻辑僵化”问题，本报告提供两套成熟的商业级解决方案。这些方案旨在从底层逻辑上解决 Agent 对工具的盲目依赖，提升响应速度与用户体验。

---

## 2. 方案 A：智能意图路由与动态工具注入 (Smart Routing & Dynamic Injection)
**核心思想：** 在模型感知到工具之前，通过确定性的逻辑或轻量级预判，物理上限制工具的暴露范围。

### 2.1 技术实现组件
1.  **意图网关 (Intent Gateway)**:
    - 在 `runAgent` 入口处，优先使用本地启发式 + 可选极速小模型，对用户意图进行分层分类。
    - 不建议只分成“知识 / 操作 / 搜索”三类，而应至少拆成：`DIRECT_ANSWER`（直接回答）, `LOCAL_READONLY`（本地只读分析）, `LOCAL_WRITE`（需要改文件/执行命令）, `WEB_LOOKUP`（需要联网获取新信息）, `BROWSER_INTERACTIVE`（需要真实浏览器交互）。
    - 额外增加一个维度：`ADVISE` / `DIAGNOSE` / `EXECUTE`，解决“用户说了修复，但其实先想听方案”的误判。
2.  **物理过滤器 (Tool Filter)**:
    - 若判定为 `DIRECT_ANSWER`，向 LLM 传递空 `tools`；若判定为 `LOCAL_READONLY`，仅挂载读文件、搜索、Git status/diff 等只读能力。
    - 将 `browser_search` 与 `browser_open/click/type` 拆成两个等级，避免“一旦命中 research，就整套浏览器交互工具全开”。
    - 消除模型“为了用工具而用工具”的心理暗示，同时保留真正需要的最小能力集。
3.  **软性证据政策 (Soft Evidence Policy)**:
    - 废除“无工具调用即失败”的截断逻辑。
    - 引入 **回答类型感知**：如果本轮目标是“解释、诊断、方案建议”，就不要求工具证据；只有当任务被判定为“已经发生真实改动/安装/配置”时，才要求工具证据。
    - 引入 **内容质量评估器**：检查回答中是否包含原因分析、解决步骤、风险提示、下一步建议。如果质量达标，即便未执行工具，也视为任务完成。
4.  **Prompt-Tool 一致性约束**:
    - `buildSystemPrompt` 只能描述“本轮实际已挂载的能力”，不要因为设置项开启了浏览器，就在每一轮都写入“Prefer browser_* tools”。
    - 把“优先使用工具”改成条件式指令，例如：仅当内部知识不足、需要最新信息、需要访问当前网页或需要真实执行时再使用工具。
5.  **浏览器预算与停止条件 (Budget & Stop Rules)**:
    - 为网页研究设置预算，例如最多一次 `browser_search`，最多一次外部跳转。
    - 当本地 workspace、Git 信息或模型已有知识已足够回答时，立即停止外部搜索。

### 2.2 商业优势
- **极速响应**：避免了重型工具（如浏览器）不必要的启动时间。
- **成本优化**：减少了无效的工具调用产生的 Token 和计算开销。
- **稳定性**：通过确定性的路由减少了模型的随机性。
- **用户体验更自然**：面对常见问答、报错解释、方案咨询时，优先给出答案，而不是先“摆出一堆工具动作”。

## 3. 方案 B：规划-执行-分析 编排架构 (Plan-Execute-Review Orchestrator)
**核心思想：** 将 Agent 从单一的“对话者”重构为“任务调度系统”，分阶段处理复杂需求。

### 3.1 技术实现流程
1.  **规划层 (Planner)**:
    - 模型第一轮交互仅允许输出 `Plan`（JSON 格式）。
    - Plan 必须包含步骤描述、所需工具等级、停止条件，以及“这是回答型步骤还是执行型步骤”的标记。
2.  **控制层 (Controller)**:
    - 由系统代码解析 Plan。
    - 如果所有步骤都是回答型 / 只读型，则立即通过 LLM 生成直接回答，流程终止。
    - 如果步骤需要工具，也要按等级逐层放开，而不是一次性挂满所有能力。
3.  **审查层 (Reviewer)**:
    - 在工具执行完后，由另一组 Prompt 或模型检查：工具输出是否解决了用户问题。
    - 如果已解决，则不再进行后续的搜索，防止“翻来覆去搜个没完”。
4.  **执行预算器 (Execution Governor)**:
    - 为每类工具设置最大调用次数、最大连续失败次数和总耗时预算。
    - 当达到预算仍未得到新信息时，强制结束工具链并回到总结回答阶段。

### 3.2 商业优势
- **极高的确定性**：适用于严谨的工程、自动化测试场景。
- **任务可见性**：用户可以清晰看到“思考中 -> 规划中 -> 执行中”的每一步进度。
- **长链条任务处理能力**：能够处理需要多点协作的复杂任务。

---

## 4. 方案对比与落地建议

| 维度 | 方案 A (智能路由) | 方案 B (全任务编排) |
| :--- | :--- | :--- |
| **首字延迟 (TTFT)** | **最低 (优秀)** | 较高 (需先生成计划) |
| **首选场景** | 桌面端助手、编程辅助、日常问答 | 自动化运维、复杂代码大规模重构 |
| **可维护性** | 高 (逻辑清晰) | 中 (状态机复杂度高) |
| **落地风险** | **低，可渐进上线** | 较高，需要重构状态流 |
| **商业案例** | **GitHub Copilot, Cursor** | **GPT-Engineer, Devin** |

### 5. 针对本项目 (Desk-Agent) 的落地优先级策略
鉴于当前项目是桌面端侧工具，**强烈建议优先实施方案 A**：

1.  **Step 0（低风险热修）:** 先修正 `buildSystemPrompt`，只描述本轮真实挂载的高级工具；同时移除或弱化全局性的 `Prefer browser_* tools` 指令。
2.  **Step 1:** 重构 `capabilitySelector.mjs`。引入“建议 / 诊断 / 执行”与“本地 / 联网 / 浏览器交互”两个维度的轻量分类，并把工具暴露改成分层注入。
3.  **Step 2:** 优化 `agent.mjs` 中的 `enforceEvidencePolicy`。只对真实执行型结果启用强证据要求，不要拦截诊断型与方案型回答。
4.  **Step 3:** 为浏览器和外部搜索加入预算、停止条件和降级策略，优先回退到直接回答或本地只读工具。
5.  **Step 4:** 补充观测指标，至少跟踪：
    - 首答是否用了工具
    - 浏览器工具调用率
    - 首字延迟（TTFT）
    - 用户二次追问“你为什么去搜网页”的比例
    - 实际修复任务的成功率

### 6. 一个更稳妥的实施原则
无论选 A 还是 B，都建议遵循以下原则：

1.  **默认最小能力**：先不给工具，或只给只读工具。
2.  **按需升级能力**：只有证据表明需要执行或联网时，才升级到写操作或浏览器。
3.  **回答优先于表演**：用户先要的是正确、快速、可信的答案，而不是 Agent 展示自己会调多少工具。
